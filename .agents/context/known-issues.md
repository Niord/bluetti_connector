# Known Issues

- The upstream BLUETTI integration mixes Home Assistant lifecycle code with reusable transport and device logic. Workaround: extract `api/`, `model/`, `profile/`, and domain pieces from `models.py` before rebuilding UI or automation layers.
- Standalone authentication requirements are not fully documented outside Home Assistant. Verified behavior: `https://sso.bluettipower.com/oauth2/token` rejects `grant_type=password` with `unsupported_grant_type`. Workaround: harden the supported access-token and refresh-token path now, and treat direct credential bootstrap as unknown from current verified behavior.
- BLUETTI `deviceStates` can report `isBindByCurUser = "0"` with an empty `stateList` even when `/devices` already reports the same device as bound. Workaround: merge product and status data, auto-call `bindDevices` when the status payload is unbound, and refetch device status before refresh or command flows.
- States that do not expose explicit allowed values in the current BLUETTI snapshot still remain read-only in the local UI, even if the upstream platform may support them through device-specific payload rules. Workaround: keep the safe switch or select surface backend-driven for now and only widen it when a model's writable values are verified.
- Backend-owned live updates intentionally stay disabled for missing auth or unsupported `ws://` websocket settings, so the fake-gateway local smoke flow cannot exercise end-to-end live delivery. Workaround: rely on manual refresh in that flow, use the focused backend and frontend live-update regression harnesses for automated coverage, and reserve live-account verification for authenticated `wss://` behavior.
- The repository does not yet include automated verification against a real BLUETTI account. Workaround: run the documented local UI smoke flow in `README.md` with the fake gateway before testing live credentials.
